# Cryptographic Assurance: What Signatures Can and Cannot Prove

Cryptographic techniques are powerful tools for deletion assurance, but they
operate within strict limits. Understanding both what they prove and what they do
not prove is essential for anyone relying on them for compliance or trust purposes.

## SHA-256: Identity Through Hashing

A cryptographic hash function maps an input of arbitrary length to a fixed-length
output (the digest). SHA-256 produces a 256-bit (32-byte) value, typically encoded
as a 64-character hexadecimal string.

Two properties make SHA-256 suitable for file identity in a deletion receipt:

**Determinism.** The same input always produces the same output. A user who holds
the original file can compute `sha256sum original_document.pdf` and compare the
result to the `file_hash_sha256` field in the receipt. If they match, the receipt
refers to their specific file.

**Collision resistance.** It is computationally infeasible to find two different
inputs that produce the same SHA-256 output. This means a receipt cannot be forged
by substituting a different file that happens to hash to the same value.

The hash is computed over the raw bytes of the file as received, before any
processing. This is non-negotiable: a hash computed after processing reflects what
the platform produced, not what the user submitted. A hash computed after deletion
reflects nothing meaningful at all — the platform could hash an empty string and
produce a validly structured receipt with no connection to the original file. The
pre-deletion commitment is what gives the hash its evidentiary weight.

## HMAC-SHA256: Authentication Through Keyed Hashing

A hash alone proves that the receipt refers to a specific file. It does not prove
that the receipt was generated by the platform rather than constructed by anyone
who knows the file. For that, a key is required.

HMAC (Hash-based Message Authentication Code, RFC 2104) is a construction that
combines a hash function with a secret key to produce an authentication code. Only
parties holding the secret key can produce a valid HMAC — or verify one. The
mathematical structure is:

```
HMAC(K, m) = H((K' XOR opad) || H((K' XOR ipad) || m))
```

The double-hashing structure resists length-extension attacks and reduces cleanly
to the pseudorandomness of the underlying hash function. HMAC-SHA256 is
standardized in RFC 2104 and FIPS 198-1 and is implemented in every major
cryptographic library.

The receipt is signed by computing HMAC-SHA256 over the canonical JSON
serialization of all fields except the signature itself. Canonical JSON means keys
sorted alphabetically at every nesting level, no optional whitespace, and
consistent Unicode encoding — because different serializations of the same logical
document produce different byte sequences, which would produce different HMAC
values, which would cause verification to fail even though the data is identical.

**HMAC is the right choice when:** the verifier is the same party as the issuer, or
when the verifier can be given the signing key through a secure channel. Internal
audit systems, enterprise deployments with contractually established keys, and
self-hosted instances all fall into this category. HMAC is simpler, faster, and
requires no key infrastructure beyond secure storage of a random string.

**HMAC is the wrong choice when:** the verifier is a third party who cannot be
trusted with the signing key. If a user needs to verify their receipt
independently, they need the key — which means the platform must share a secret
with an untrusted party, undermining the secrecy that gives the key its value.

## ECDSA: Asymmetric Signing for Third-Party Verification

Asymmetric cryptography separates signing from verification. With ECDSA (Elliptic
Curve Digital Signature Algorithm), the private key is used to sign and the public
key is used to verify. Anyone with the public key can verify a signature without
being able to produce one. The private key never leaves the signing system.

P-256 (also known as secp256r1 or prime256v1) is the NIST-recommended curve used
in `deleteceipt`. It produces compact signatures (96–104 bytes in DER encoding,
base64-encoded to ~128 characters), is fast on modern hardware, and is supported
by every major TLS and cryptography library.

`deleteceipt` embeds the public key in the receipt itself (`signing_public_key_pem`
field). This makes the receipt self-contained: any verifier who receives the receipt
can verify the signature without contacting the server. The server's `/public-key`
endpoint allows out-of-band verification that the embedded key matches the server's
claimed identity.

**ECDSA is the right choice when:** deletion proof must be presented to a user,
regulator, or auditor who cannot be trusted with a symmetric key. The receipt is
portable evidence that stands on its own.

**Non-repudiation as a double edge.** Asymmetric signatures are often valued for
non-repudiation — the signer cannot plausibly deny having signed. In deletion
contexts, this cuts both ways. A signed deletion record is evidence that the
organization performed the deletion, but it is also evidence that the organization
was aware the data existed, held it in a deletable form, and made a specific
decision to delete it at a specific time. In some regulatory contexts, this
documentation may create legal exposure the organization prefers to avoid.

## Hash Chains: Tamper-Evident Audit Logs

A hash chain makes an append-only log tamper-evident. Each entry includes a
cryptographic hash of the previous entry. To modify entry N retroactively, an
attacker must also recompute entry N+1 (which covers N's hash), and N+2, and every
subsequent entry — a cascade of invalidated hashes that is detectable upon
verification.

```
entry_0: {event: "upload",   prev_hash: "0"*64,        ... , entry_hash: "abc..."}
entry_1: {event: "delete",   prev_hash: "abc...",       ... , entry_hash: "def..."}
entry_2: {event: "receipt",  prev_hash: "def...",       ... , entry_hash: "ghi..."}
```

`verify_chain()` recomputes each entry's hash from its contents and checks that it
matches the `prev_hash` of the following entry. A broken link — where the computed
hash does not match the recorded `prev_hash` — indicates tampering.

**One critical caution: hash chains prove tamper-evidence, not completeness.** A
hash chain tells you that the entries it contains have not been modified since the
chain was initialized. It does not tell you that the chain contains all deletion
events that should have been recorded. Entries that were never added to the chain —
deletion events that the workflow failed to record, categories of data that were
not covered by the deletion workflow — are invisible to the chain's integrity
verification.

Complete deletion assurance requires both a sound deletion workflow (so that all
deletions are recorded) and hash-chained logs (so that recorded deletions cannot be
suppressed).

Periodic anchoring strengthens the guarantee: publishing the most recent chain hash
to an externally controlled, append-only location — a separate database, a
third-party log service, a public repository — means that to suppress a deletion
record, an attacker must rewrite not just the local log but also all external
anchors published after the record was written.

## What Cryptographic Signatures Cannot Prove

A deletion audit log signed with HMAC-SHA256 proves that the deletion workflow
produced a record stating that a specific piece of data was processed for deletion
at a specific time by a specific actor, and that this record has not been modified
since it was signed.

It does not prove:

- That the deletion workflow actually removed the data from all storage locations
- That the workflow had access to all copies of the data (including those in systems
  it was not integrated with)
- That the workflow was correctly implemented (signing operates over the output of
  the workflow, not over its internal correctness)
- That no copies of the data exist outside the signing system's knowledge
- That the storage paths in the manifest (`/mnt/processing/...`) were actually
  backed by the storage medium the paths suggest

The signature on the audit log does not say "this data has been deleted everywhere."
It says "this data was processed by this workflow." The gap between those two
statements is where legal and technical risk accumulates.

The correct frame for cryptographic deletion assurance is probabilistic and layered,
not absolute. Cryptographic techniques provide strong evidence that deletion
workflows were executed correctly on the systems they were designed to cover. That
evidence is meaningful and useful for compliance demonstrations, regulatory
responses, and user trust. It is not a guarantee of perfect, universal,
unrecoverable deletion in the face of all possible threat models.

Building systems that make that guarantee honestly requires combining cryptographic
evidence with architectural discipline (data isolation, TTL enforcement, data
minimization, ephemeral storage for transient data), operational rigor (complete
data inventories, third-party vendor data processing agreements, developer access
controls on production data), and procedural verification (periodic audits of
deletion workflow coverage against the data inventory).

Cryptography secures the evidence. The architecture secures the data. Both are
required, and neither is sufficient alone.

## Choosing Between HMAC and ECDSA

| Property | HMAC-SHA256 | ECDSA / P-256 |
|----------|-------------|---------------|
| Key type | Shared secret | Private / public keypair |
| Who can verify | Anyone with the secret key | Anyone with the public key |
| Key distribution risk | Must share secret with verifier | Public key only — no secret shared |
| Self-contained receipt | No | Yes (`signing_public_key_pem` embedded) |
| Non-repudiation | No (both parties hold the key) | Yes (only signer holds private key) |
| Performance | ~orders of magnitude faster | Slower but still fast in practice |
| Key management complexity | Simpler | Requires private key lifecycle management |
| Best for | Internal systems, enterprise self-hosted | User-facing receipts, third-party auditors |

Use HMAC when the verifier is the same party as the issuer. Use ECDSA when you
need users or regulators to verify receipts independently without possessing any
secret key.
